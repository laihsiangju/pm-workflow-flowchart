PM Workflow / 詳細流程拆解

三個 AI 專案 × 逐步操作細節 — 前期、中期、後期

📋
前期 / Before — Proposal
Proposal Creator
Proposal_Creator/ · /brainstorm-proposal
1
PM 輸入
PM 輸入客戶需求
口頭釐清客戶需求、討論範圍與方向，作為後續分析的起點。
💬 /brainstorm-proposal 客戶名稱
2
AI 自動
讀合作案 + PRD + Codebase（並行）
Drive 找「客戶名 + 合作案」文件、讀 TMS / App / InstaTMS PRD、用 Agent 搜 Codebase 既有實作。
📁 Drive 📄 Docs 💻 Codebase
3
AI + PM
釐清需求（一次一題）
優先用選擇題、聚焦需求目的 / 限制條件 / 優先級。

AI 自動
需求合併 + 標註已有實作
區分「已完成（無需開發） / 可複用（部分開發） / 全新（從零開發）」。
5
AI 自動
產出三方案
產品化 / 中間 / 暴力解，列出各方案描述與與現有功能的關係。

審核草稿、選定方案
PM 檢視三方案、選定後確認是否寫入合作案。

AI 自動
寫回合作案表格
用 Google Docs REST API + OAuth：insertTableRow 新增列，reverse index 填各欄位（非 MCP）。

/create-jira-ticket（可選）
對話內容即可直接轉 Jira Story：一次 call 建票、description 帶 ADF、AC 4 層 bullet。建票前列出 Story 等 PM 確認。

中期 A / During — Jira Story
Story Generator (建立 Jira Story)
Story&Spec_Generator/ · generate-story
1
PM 輸入
觸發「產生 Story，[客戶名]」
提案通過後 PM 下觸發詞，skill 直接執行、不問額外問題。
💬 skill: generate-story
2
AI 自動
Drive 搜客戶合作案
命名慣例：「客戶名稱 + 合作案」。若 mimeType 為 shortcut，必須用 mimeType 重搜取實際文件 ID。
📁 Drive
3
AI 自動
擷取「客製化功能」章節表格
每列作為一個獨立功能點進行後續分析。
📄 Docs
4
AI 自動
轉 Story — AC 層次化 bullet (4 層)
Summary：身為 [角色]，我想要 [功能]，以便於 [價值]。
AC：明令不用 Given/When/Then；含「新增欄位連動檢查」規則（派車單匯出、訂單報表匯出要補欄位）。
🤖 Claude
5
PM 審閱
人工審核（嚴禁直接開票）
把所有 Story 列在 Chat，等 PM 說「確認」或「OK」才建票。
6
AI 自動
查既有 Epic 當 Parent
能對應才帶 parent_key；判不出來整個欄位不帶（不要猜、不要 null）。
🎫 Jira MCP
7
AI 自動
建 Jira Story（description 直接帶 ADF）
一次 call 完成：情境（粗體+底線）+ 巢狀 bulletList（4 層）對應 AC。
🎫 Jira MCP
🔄
Sprint 開發過程 · AC 多次人工調整
開發進行中，PM 會因實際情境、技術限制、客戶回饋等原因，多次手動修訂 Jira Story 的 AC 與描述 （mcp-atlassian 沒有 update issue 工具 → 用 curl PUT 打 Jira REST API 塞 ADF）。 待 AC 定稿後才將最終版本回寫 PRD。
🎫 Jira Iteration 🧑 PM 手動調整
📘
中期 B / During — 寫入 PRD
Spec Generator (回寫 PRD)
Story&Spec_Generator/ · write-spec
1
PM 輸入
觸發二選一（skill 開頭先判斷模式）
🅰️ 單票模式：訊息含 DAAS-XXXX URL 或 key
🅱️ 批次模式：「整理規格寫入規格書」/「寫入 PRD」/「open sprint 的票寫入規格書」/「整個 sprint」
兩者都不明確 → 直接問用戶
💬 skill: write-spec
2
AI 自動
讀 Jira ticket(s)
🅰️ 單票：jira_get_issue 讀該票（fields: *all）。
🅱️ 批次：若沒指定 Sprint → jira_get_sprints_from_board（board_id=38, state=active）取 openSprint 並向用戶確認「要寫入 Sprint X 的全部 ticket 嗎？」→ jira_search 完整分頁撈完；只保留 Story/Task（Bug 不寫 PRD）；排除 QA/設計/Infra/E2E/PRD 關鍵字。
🎫 Jira MCP
3
AI 自動
依產品線分組 + 整理 4 章節
summary tag [TMS]/[App]/[InstaTMS] 分流對應 PRD Doc；無 tag → 單獨列出讓用戶決定。
4 章節：功能概述 / 使用者價值 / 使用場景 / 功能列表。不含成功指標、UI/UX 要點。
🅱️ 批次：每張 ticket 各自整理。
🤖 Claude
4
AI 自動
gdocs-inspect + 規劃插入位置
🅰️ 單票：取 heading 結構與 index、建議章節編號與插入位置。
🅱️ 批次：每份 PRD 跑一次 inspect，一次規劃該 PRD 內所有待寫票位置，並按 startIndex 由大到小排序寫入順序（從文件底部往上插，避免 index 偏移）。
💻 gdocs-inspect.js
5
PM 審閱
展示規劃 + 等待確認
把整理好的完整內容 + 建議位置展示給用戶，等用戶確認後才寫入。
🅱️ 批次：一次列出每張票要寫入哪份 PRD、哪個章節、寫入順序；用戶可逐張取消或修改，但只做一輪確認（不逐張各問）。
6
AI 自動
自寫 Node 腳本寫入 PRD Doc
用 batchUpdate API 精準定位；禁用 MCP replaceText；只能新增、不可移除。
格式比照鄰近章節（HEADING_3/4、bold、bullet）。
🅱️ 批次：同 PRD 內 reverse index order；跨 PRD 逐個 Doc 處理。
📄 Docs API
7
AI 自動
更新 Doc 頂部「版本紀錄檔案」表格
日期 / SP / 說明 / 異動人員；同 Sprint 已有條目用 replaceAllText 追加到現有列（不新增一列）。
🅱️ 批次：每份 PRD 只新增/追加一筆，說明欄合併列出所有票號（例「新增 3.1.3.3 / 3.1.3.4（DAAS-6886, DAAS-6887）」）。
8
AI 自動
驗證格式 + 清理暫時腳本
再跑一次 gdocs-inspect + 詳細檢查 bold / bullet / heading 與鄰近既有章節一致，最後刪除本次產生的暫時腳本。
🚀
後期 / After — Release
TMS Release Note
TMS_Release_Note/ · release-note
1
PM 輸入
觸發「產生 Release Note」/「go」
Sprint 接近收尾，PM 下觸發詞，skill 立刻依序執行、不問問題。
💬 skill: release-note
2
AI 自動
撈 Jira openSprints（完整分頁）
JQL: project = DAAS AND sprint in openSprints()，limit 50；必須檢查 next_page_token 撈完（過去漏過第二頁的 Story）。
🎫 Jira MCP
3
AI 自動
過濾 Story / Task / Bug
summary 不含 QA / 設計 / Infra / E2E / jenkins / docker / 腳本 / PRD / Functional·UI test。
4
AI 自動
讀 Doc 最新 HEADING_1 + 14 天
用 curl 打 Docs API，只解析 body.content 前幾個 paragraph（不用 docs_getText 讀整份，省 tokens）。
📄 Docs API
5
AI 自動
分類 + 撰寫風格固化
Story / Task → 新增 or 優化；Bug → 一律進優化＆修復。[TMS]/[APP]/[前端]/[後端] 決定平台。
風格：功能名稱精煉、用戶角度用詞、「名詞+修正」語氣、2-3 句短句「功能是什麼→價值」、描述不重複標題。
🤖 Claude
6
AI 自動
寫 draft.md（4 header 強制）
★ 新增功能 / ◉ 優化＆修復 × TMS / APP 共 4 個 header，缺項也要留標題 + 「無」。
不加 */- bullet、不加 markdown 粗體（腳本會自動套 Doc 樣式）。
💻 draft.md
7
PM 審閱
編輯 draft.md → 說「寫入」
PM 調整客戶端可見描述、補公告或已知問題；說「寫入」後才推進下一步。
8
AI 自動
write_to_gdoc.py 寫入 Doc
專屬 Python 腳本自動處理：找 Doc 最新 HEADING_1、插入新內容、套 14pt / 粗體 / bullet / pageBreak。格式規則全在腳本，不手動 batchUpdate。
💻 write_to_gdoc.py 📄 Docs
9
AI 自動
Google Chat 通知（格式固化）
板 id=38 取 active sprint 名 → space spaces/AAAALHxo-B4：
<users/all> {Sprint 名} Release Note 已更新 ✅ + 日期 + 本次聚焦（新增功能串接）+ Doc 連結。
💬 Chat
🧑 PM 介入的關鍵節點
  • 口頭輸入客戶需求 → 啟動 Proposal
  • 釐清需求（一次一題）+ 選定三方案
  • Story 草稿審核（嚴禁直接開票）
  • Sprint 中 AC 多次手動調整（curl 塞 ADF）
  • PRD 寫入確認：單票 / Sprint 整批
  • Release Note draft.md 用字調整
🤖 AI 自動化的重點範圍
  • 並行抓資料（Drive / PRD / Codebase）
  • 需求合併 + 既有實作比對（已完成/可複用/全新）
  • Jira 建票 description 直接帶 ADF（一次 call）
  • PRD 寫入用 Docs REST API（禁 MCP replaceText）
  • Release Note 自動分頁撈、自動寫 Doc、自動通知 Chat
🔄 三階段資料流向
  • 前期：需求 + 合作案 + Codebase + PRD → 合作案表格 / Jira Story
  • 中期 A：合作案「客製化功能」表 → Jira Story（ADF）
  • Sprint：Jira AC 反覆調整（curl PUT）
  • 中期 B：Jira ticket / Sprint → 回寫 PRD（TMS/App/InstaTMS）
  • 後期：openSprints → Release Note Doc + Chat 通知
