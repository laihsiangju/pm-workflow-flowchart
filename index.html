PM Workflow / 詳細流程拆解

三個 AI 專案 × 逐步操作細節 — 前期、中期、後期

📋
前期 / Before — Proposal
Proposal Creator
Proposal_Creator/
1
PM 輸入
接收客戶需求
業務或客戶提出新專案需求，PM 彙整背景資訊與初步目標。
2
AI 自動
搜尋歷史提案
掃描 Drive 客戶資料夾，找出類似產業 / 客戶過往提案作參考。
📁 Drive
3
AI 自動
讀取既有文件
讀取 Google Docs 中現有的客戶資料、合約、會議紀錄等。
📄 Docs
4
AI 自動
掃描前端 Codebase
讀取 /tms/ 與 /app/ 原始碼，確認既有功能模組與可重用元件。
💻 tms/ 💻 app/
5
AI 自動
解析 API 定義
讀取 /api定義/ 文件，比對客戶需求與現有 API 覆蓋範圍。
💻 api定義/
6
AI 自動
產出提案草稿
依據客戶需求與既有資源，生成提案架構、範疇、時程與風險。
🤖 Claude
7
PM 審閱
人工修訂與調整
PM 檢視草稿、補上商業判斷與客戶洞察，調整用字與細節。
8
AI 自動
寫入 Google Docs
最終版本建立 / 更新至 Google Docs，分享給客戶與內部相關方。
📄 Docs
🎫
中期 A / During — Jira Story
Story Generator (建立 Jira Story)
Story&Spec_Generator/ · Story
1
PM 輸入
提案通過 / 啟動開發
客戶簽約後，PM 開始將提案轉為可執行的 User Story。
2
AI 自動
搜尋客戶資料夾
至 Drive 定位該客戶專案資料夾，抓取客製化功能表。
📁 Drive
3
AI 自動
讀取客製化需求
讀取 Google Docs 客製化功能文件，擷取功能清單與接受準則。
📄 Docs
4
AI 自動
查詢 Jira Epic
於 Jira 中尋找既有 Epic，作為新 Story 的 Parent Ticket。
🎫 Jira
5
AI 自動
參考 Figma 設計稿
連動 Figma，讀取 UI 流程與畫面，補充至 Story 描述中。
🎨 Figma
6
AI 自動
拆分 User Story
以客戶功能為單位拆成多張 Story，產生 初版 AC、描述、估點建議。
🤖 Claude
7
PM 審閱
Story 初版驗收
PM 檢查 Story 切分是否合理、優先順序與初版 AC，補缺漏情境。
8
AI 自動
建立 Jira Stories
批次建立 Jira Story（掛在 Epic 底下），正式進入開發 Sprint。
🎫 Jira
🔄
Sprint 開發過程 · AC 多次人工調整
開發進行中，PM 會因實際情境、技術限制、客戶回饋等原因，多次手動修訂 Jira Story 的 AC 與描述。 待 Sprint 驗收完畢、AC 定稿後，才會啟動下一階段產出最終 Spec。
🎫 Jira Iteration 🧑 PM 手動調整
📐
中期 B / During — Product Spec
Spec Generator (建立 Product Spec)
Story&Spec_Generator/ · Spec
1
PM 輸入
Sprint 驗收完畢
開發完成、所有 Story 驗收通過，最終版 AC 已在 Jira 上定稿。
2
AI 自動
抓取最終 Jira Stories
重新讀取已驗收完成的 Jira Story（含 Sprint 中多次修訂後的 AC）。
🎫 Jira
3
AI 自動
比對 Figma 最終稿
同步最新 Figma UI，確保 Spec 畫面描述與實作一致。
🎨 Figma
4
AI 自動
OpenSpec 結構化
以 OpenSpec 標準格式整合 Stories + AC + Figma，輸出 Product Spec 草稿。
📐 OpenSpec 🤖 Claude
5
PM 審閱
Spec 內容檢查
PM 檢查 Spec 是否完整對應最終實作、補充說明與備註。
6
AI 自動
寫入產品規格書
最終 Spec 歸檔至產品規格書（Google Docs / Confluence），供後續維護參考。
📄 Docs 📐 Spec
🚀
後期 / After — Release
TMS Release Note
TMS_Release_Note/
1
PM 輸入
Sprint 接近收尾
開發 Sprint 進入結束階段，準備整理本次版本的交付內容。
2
AI 自動
查詢 Open Sprint
連接 Jira DAAS 專案，撈取當前 Open Sprint 的所有 Tickets。
🎫 Jira
3
AI 自動
過濾已完成 Tickets
只保留 Done / Released 狀態，排除未完成或延至下期的項目。
4
AI 自動
讀取歷史 Release Note
從 Google Docs 讀取前幾個版本的 Release Note，推算本次版號與日期。
📄 Docs
5
AI 自動
分類 Tickets
依 Feature / Enhancement / Bug Fix 分類，整理成 Release Note 結構。
🤖 Claude
6
AI 自動
產出 draft.md
將整理好的內容寫入本地 draft.md，供 PM 先行審閱。
💻 draft.md
7
PM 審閱
人工調整用字
PM 調整客戶端可見的描述用字、補充重要公告或已知問題。
8
AI 自動
寫入 Google Docs
最終版本格式化後寫入 Google Docs Release Note 集中頁面。
📄 Docs
9
AI 自動
自動通知 Chat 群組
Release Note 寫入 Google Docs 後，自動觸發訊息發送至 Google Chat 群組，通知客戶與內部團隊本次版本更新。
💬 Chat
🧑 PM 介入的關鍵節點
  • 接收客戶需求 → 啟動 Proposal
  • 提案草稿 → 商業判斷修訂
  • Story 初版驗收 → 建立 Jira
  • Sprint 中 AC 多次人工調整（關鍵）
  • Sprint 驗收完 → Spec 內容檢查
  • Release Note → 用字調整（公告已自動化）
🤖 AI 自動化的重點範圍
  • 跨系統資料彙整（Drive、Jira、Figma）
  • Codebase 與 API 的一致性比對
  • 結構化文件生成（Stories、Spec、Notes）
  • 自動寫回 Jira / Google Docs
  • Release Note 發佈後自動通知 Chat 群組
🔄 四階段資料流向
  • 前期：Drive + Codebase → Docs 提案
  • 中期 A：Docs + Figma → Jira Stories
  • Sprint：Jira AC 人工反覆調整
  • 中期 B：定稿 Jira → Product Spec
  • 後期：Jira Sprint → Docs Release Note
