PM 主導
Sprint 啟動、工程開發進行中

🧑 PM 手動
3

中後期 / Post-Acceptance
Spec Finalization（驗收後寫入 PRD）
Spec_Finalization/ · /write-spec · /write-spec-batch
1
PM 輸入
觸發：兩個 skill 二選一
🅰️ /write-spec（單票）：訊息含 DAAS-XXXX URL 或 key
🅱️ /write-spec-batch（批次）：「整理規格寫入規格書」/「寫入 PRD」/「open sprint 的票寫入規格書」
時機：sprint 驗收後，非開發前 spec。
💬 /write-spec 💬 /write-spec-batch
2
AI 自動
讀 Jira ticket(s)
🅰️ 單票：jira_get_issue 讀該票（fields: *all）。
🅱️ 批次：若沒指定 Sprint → 取 openSprint → jira_search 完整分頁撈完；只保留 Story/Task（Bug 不寫 PRD）；排除 QA/設計/Infra/E2E/PRD 關鍵字。
🎫 Jira MCP
3
AI 自動
依產品線分組 + 整理 4 章節
summary tag [TMS]/[App]/[InstaTMS] 分流對應 PRD Doc；無 tag → 單獨列出讓用戶決定。
4 章節：功能概述 / 使用者價值 / 使用場景 / 功能列表。
🤖 Claude
4
AI 自動
gdocs-inspect + 規劃插入位置
🅰️ 單票：取 heading 結構與 index、建議章節編號與插入位置。
🅱️ 批次：每份 PRD 跑一次 inspect，一次規劃該 PRD 內所有待寫票位置，並按 startIndex 由大到小排序寫入順序（從文件底部往上插，避免 index 偏移）。
💻 gdocs-inspect.js
5
PM 審閱
展示規劃 + 等待確認
把整理好的完整內容 + 建議位置展示給用戶，等用戶確認後才寫入。
🅱️ 批次：一次列出每張票要寫入哪份 PRD、哪個章節、寫入順序；用戶可逐張取消或修改，但只做一輪確認。
6
AI 自動
自寫 Node 腳本寫入 PRD Doc
用 batchUpdate API 精準定位；禁用 MCP replaceText；只能新增、不可移除。
格式比照鄰近章節（HEADING_3/4、bold、bullet）。
📄 Docs API
7
AI 自動
更新 Doc 頂部「版本紀錄檔案」表格
日期 / SP / 說明 / 異動人員；同 Sprint 已有條目用 replaceAllText 追加到現有列（不新增一列）。
8
AI 自動
驗證格式 + 清理暫時腳本
再跑一次 gdocs-inspect + 詳細檢查 bold / bullet / heading 與鄰近既有章節一致，最後刪除本次產生的暫時腳本。

後期 / Release
TMS Release Note
TMS_Release_Note/ · release-note
1
PM 輸入
觸發「產生 Release Note」/「go」
Sprint 接近收尾，PM 下觸發詞，skill 立刻依序執行、不問問題。
💬 skill: release-note
2
AI 自動
撈 Jira openSprints（完整分頁）
JQL: project = DAAS AND sprint in openSprints()，limit 50；必須檢查 next_page_token 撈完（過去漏過第二頁的 Story）。
🎫 Jira MCP
3
AI 自動
過濾 Story / Task / Bug
summary 不含 QA / 設計 / Infra / E2E / jenkins / docker / 腳本 / PRD / Functional·UI test。
4
AI 自動
讀 Doc 最新 HEADING_1 + 14 天
用 curl 打 Docs API，只解析 body.content 前幾個 paragraph（不用 docs_getText 讀整份，省 tokens）。
📄 Docs API
5
AI 自動
分類 + 撰寫風格固化
Story / Task → 新增 or 優化；Bug → 一律進優化＆修復。[TMS]/[APP]/[前端]/[後端] 決定平台。
風格：功能名稱精煉、用戶角度用詞、「名詞+修正」語氣、2-3 句短句「功能是什麼→價值」、描述不重複標題。
🤖 Claude
6
AI 自動
寫 draft.md（4 header 強制）
★ 新增功能 / ◉ 優化＆修復 × TMS / APP 共 4 個 header，缺項也要留標題 + 「無」。
不加 */- bullet、不加 markdown 粗體（腳本會自動套 Doc 樣式）。
💻 draft.md
7
PM 審閱
編輯 draft.md → 說「寫入」
PM 調整客戶端可見描述、補公告或已知問題；說「寫入」後才推進下一步。
8
AI 自動
write_to_gdoc.py 寫入 Doc
專屬 Python 腳本自動處理：找 Doc 最新 HEADING_1、插入新內容、套 14pt / 粗體 / bullet / pageBreak。格式規則全在腳本，不手動 batchUpdate。
💻 write_to_gdoc.py 📄 Docs
9
AI 自動
Google Chat 通知（格式固化）
板 id=38 取 active sprint 名 → space spaces/AAAALHxo-B4：
<users/all> {Sprint 名} Release Note 已更新 ✅ + 日期 + 本次聚焦（新增功能串接）+ Doc 連結。
💬 Chat
🧑 PM 介入的關鍵節點
  • 起點：客戶案 /shape-proposal vs 產品化題目（直接討論）
  • 釐清需求（一次一題）+ 方案審核
  • 客戶案：等簽約後才繼續 /kickoff-from-proposal
  • 建票前列 Story 審核（嚴禁直接開票）
  • Sprint 中 AC 多次 curl PUT 手動調整
  • 驗收後：/write-spec（單票）或 /write-spec-batch（批次）寫入確認
  • Release Note：draft.md 用字調整 → 說「寫入」
🤖 AI 自動化範圍
  • 並行抓資料（Drive / PRD / Codebase / Figma）
  • 客戶案：三方案產出 + 合作案回寫
  • 產品化題目：PM 提供資料 → 共同討論形塑
  • Jira 建票 description 直接帶 ADF（一次 call）
  • PRD 寫入用 Docs REST API（禁 MCP replaceText）
  • Release Note 自動分頁撈、寫 Doc、通知 Chat
🔄 四階段資料流向
  • P1 Shape & Kickoff：需求 + 合作案 + Codebase → 方案 / Jira Story
  • P2 Sprint：AC 反覆調整（curl PUT 塞 ADF）→ 驗收完成
  • P3 Spec Finalization：驗收 Ticket → PRD（TMS/App/InstaTMS）
  • P4 Release Note：openSprints → Doc + Chat 通知
