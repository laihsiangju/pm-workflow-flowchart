1 需求形塑 Shape Kickoff

目標： 從客戶需求釐清到提案產出。支援兩種情境：客戶案（讀合作案文件）與內部案（直接討論），透過需求分析、功能匹配、三方案比較，產出結構化提案。
完成後： 提案已確認、已寫入合作案文件（客戶案）或準備開 Jira ticket（內部案）。

1a客戶提案腦力激盪

探索客戶需求、匹配現有功能、產出產品化/中間/暴力解三種方案
資料來源 / INPUT
📁 Drive 📄 PRD 💻 Codebase
Codebase 由固定腳本每雙週固定時間自動撈取 Bitbucket 內容，並更新本地資料夾。
📌 使用 Skill：
/shape-proposal 客戶名稱
✓ 輸出內容：
  • 客製化需求三方案清單
  • 既有功能直接提供清單
  • 受影響的既有模組列表

1bHTML 原型設計

依 Figma 設計系統產出流程 prototype，視覺 100% 貼齊設計系統（TMS 後台 / APP 手機端）。
資料來源 / INPUT

✓ 輸出內容：
  • HTML 互動原型
  • 完整設計系統元件
  • 頁面流程展示

1c寫入合作案文件

把確認後的客製化方案寫入客戶合作案的「客製化功能」表格（功能欄位，估時留空）。
資料來源 / INPUT
📁 Drive 📄 提案草稿
📌 使用 Skill：
/save-proposal
✓ 輸出內容：
  • 合作案文件已更新
  • 客製化功能表格新增列
  • 方案描述已記錄
💡 重點提醒：
  • 客戶案需在簽約後才進入下一階段（Jira 開票）
  • 內部案可以跳過「寫入合作案」步驟，直接開 Jira ticket
  • 原型設計是選配，視客戶需求而定

2 啟動執行 Kickoff

目標： 把已確認的需求轉譯為 Jira Story ticket。支援兩種轉譯來源：對話內容（內部案）或客戶合作案文件（客戶案）。
完成後： Jira Story ticket 已建立，開發團隊可開始規劃與實作。

2a從對話轉譯開票

從當前對話中已確認的功能需求，轉譯為 Jira Story 格式（含 ADF description），建立 ticket。適合內部案。
資料來源 / INPUT
💬 Chat
📌 使用 Skill：
/kickoff-from-chat
✓ 輸出內容：
  • Story 標題格式（身為/我想要/以便於）
  • 層次化 Acceptance Criteria
  • Jira ticket ID（DAAS-XXXX）

2b從合作案轉譯開票

從客戶合作案文件的「客製化功能」表格，逐列轉譯為 Jira Story，批次建立 ticket。適合客戶案。
資料來源 / INPUT
📁 Drive 📄 合作案
📌 使用 Skill：
/kickoff-from-proposal 客戶名稱
✓ 輸出內容：
  • Story 標題格式（身為/我想要/以便於）
  • 層次化 Acceptance Criteria
  • 批次 Jira ticket ID（DAAS-XXXX，DAAS-XXXX,...）
💡 Story 撰寫規則：
  • 標題格式： 「身為 [角色]，我想要 [功能]，以便於 [價值]」
  • AC 格式： 層次化 bullet（最多 4 層）
  • 欄位連動： 若新增欄位會在訂單報表出現，AC 必須標註派車單/訂單報表匯出變更
  • 簽約後開票： 客戶案須在用戶簽約並確定實作後，才建立 Jira ticket

3 規格定案 Spec Finalization

目標： Sprint 驗收完成後，整理產品規格文件。從已驗收的 Jira ticket 提取核心 AC，整理為 4 章節規格（功能概述/使用者價值/使用場景/功能列表），寫入對應產品線的 Google Doc PRD。
完成後： PRD 文件已更新，包含完整的功能規格與版本紀錄。

3a單票整理規格

提供單一 Jira ticket，整理為 4 章節格式，寫入對應 PRD。適合零星驗收或需特殊處理的票。
資料來源 / INPUT
🎫 Jira 📄 PRD
📌 使用 Skill：
/write-spec
✓ 輸出內容：
  • 4 章節規格內容
  • Google Doc PRD 已更新
  • 版本紀錄已新增

3b批次整輪規格

整個 Sprint 的所有 Story/Task 一次性整理，依產品線（TMS/App/InstaTMS）分組，批次寫入各自的 PRD。
資料來源 / INPUT
🎫 Jira 📄 PRD
📌 使用 Skill：
/write-spec-batch
✓ 輸出內容：
  • 每份 PRD 的 4 章節規格
  • 三份 Google Doc（TMS/App/InstaTMS）已更新
  • 版本紀錄彙整統一
💡 規格整理規則：
  • 觸發時機： Sprint 末尾、驗收都確認 OK 後才啟動
  • 4 章節格式： 功能概述（1段）→ 使用者價值（列點）→ 使用場景（誰、何時）→ 功能列表（操作層級）
  • 章節標題連結： 標題文字指向對應 Jira ticket（只有標題，不含編號）
  • 不可移除既有： 只能新增，禁止刪除或修改已有內容
  • 版本紀錄： 記錄日期、Sprint、異動說明、異動人員

4 版本發布 Release Note

目標： 版本發布目前已全自動化。固定每兩週 Sprint Demo 後，自動撈取所有開發票務（Story/Task/Bug），產生版本發布說明（Release Note），包含新增功能、優化修復，並通知相關利益者。
完成後： Release Note 已自動發布在 Google Doc，Google Chat 已自動通知全員。

4Release Note 產生

固定每兩週 Sprint Demo 後自動產生本次版本發布說明。系統自動撈取 open sprint 的全部 Story/Task/Bug，分類為「新增功能」與「優化修復」，按產品線分組（TMS/APP），撰寫用戶友善說明，最後推送到 Google Doc 並通知 Google Chat。
資料來源 / INPUT
🎫 Jira 📄 Release Note Doc
📌 使用 Skill：
/release-note
固定排程：每兩週 Sprint Demo 後自動執行
✓ 輸出內容：
  • Release Note draft.md（專案根目錄）
  • Google Doc Release Note 已發布
  • Google Chat 通知已推送
  • 包含：本次聚焦、功能清單、日期、ticket 連結

Release Note 撰寫要點

保持用戶友善、簡潔清晰的風格。
✓ 撰寫原則：
  • 功能名稱精煉，不加「功能」二字
  • 用用戶角度用詞，避免技術術語
  • 優化修復統一用「名詞 + 修正」格式
  • 每項 2-3 句簡述，聚焦價值
  • 分為：TMS、APP 兩平台
  • 分為：「新增功能」、「優化修復」兩類別

🌳 工作流程決策樹

是否為客戶案？

✓ 客戶案： 從客戶合作案文件讀取需求 → 執行 /shape-proposal → /save-proposal → /kickoff-from-proposal

✓ 內部案： 直接討論功能 → /kickoff-from-chat

是否需要產生 Prototype？

✓ 是： 在 /shape-proposal 後執行 /build-prototype，產出 HTML 原型

✓ 否： 跳過原型步驟，直接進入提案存檔或開票

规格整理時機

✓ 單票驗收： 執行 /write-spec（零星或緊急）

✓ 整輪 Sprint： 執行 /write-spec-batch（定期，Sprint 末）

版本發布

✓ 每兩週 Sprint Demo 後： 自動執行 /release-note（自動化撈票、分類、寫 doc、推送通知）
